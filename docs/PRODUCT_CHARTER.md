# Project Charter

> **Home Energy Advisor** *(working title)*

## Purpose

Home Energy Advisor exists to help Home Assistant users understand the financial impact of their energy consumption and make better energy decisions.

The product should present meaningful information, not just more data. Every feature should help users understand their energy usage, reduce costs or make better-informed decisions.

## Success

This product succeeds when users can make better energy decisions more easily.

## Principles

### Deliver Value

Every release should provide tangible value to users. We prioritise outcomes over features and avoid work that does not improve the user experience or enable future user value.

### Simplicity Over Cleverness

Choose the simplest solution that solves today's problem well. Optimise when there is evidence that optimisation is needed, not because it might be useful in the future.

### Evolve Through Learning

Design the product so that it can evolve as we learn from real users. Avoid predicting future requirements or introducing unnecessary abstraction before it has demonstrated value.

### Documentation Explains Intent

Documentation exists to capture design intent, important decisions and the rationale behind them.

Documentation should not duplicate information that is already obvious from the implementation.

### Code Explains Implementation

Code should be written to be clear, readable and self-explanatory through good design, naming and structure.

If implementation is difficult to understand without external documentation, the implementation should be reconsidered before additional documentation is written.

### Tests Describe Behaviour

Tests are first-class documentation.

They demonstrate how the product is expected to behave through executable examples covering normal operation, edge cases and regression scenarios.

A contributor should be able to understand the expected behaviour of the system by reading the tests.

### Local First

The product should operate entirely within the user's own environment wherever practical. Cloud services should never be required unless they provide optional functionality chosen by the user.

### Open Source First

Use open standards and open source technologies wherever practical.

The project should be approachable for contributors and avoid unnecessary dependencies on proprietary tooling or services.

### Evidence Over Assumption

Make engineering decisions based on measurement, evidence and real-world experience.

Avoid adding complexity to solve problems that have not yet been demonstrated.

## Living Document

This charter defines how decisions are made within the product.

It is intentionally concise and should evolve only when our engineering principles or product philosophy change through experience.
